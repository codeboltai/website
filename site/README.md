# Codebolt Astro Cloudflare Template

This is the Codebolt website template for `@codebolt/agent-sitegen`.

It is an Astro + Tailwind static site template that can be generated from structured content and deployed to Cloudflare Workers static assets with Wrangler.

## How This Template Is Used

From the website repo root:

```bash
npm run validate
npm run generate
npm run --prefix ./site build
```

The root scripts use:

```bash
sitegen generate --template ./template --content ./content/site.yaml --output ./site
```

The generated site is written to `site/`. Treat `site/` as output. Make durable changes in `content/` or `template/`.

## Template Contract

Important files:

- `sitegen.template.json`: template metadata, generated paths, component groups, and private helper components.
- `src/components/`: Astro components that content sections can render.
- `src/layouts/BaseLayout.astro`: generated pages use this layout.
- `src/data/site-data.js`: generated by the CLI.
- `src/pages/`: generated by the CLI.
- `public/`: copied static assets.
- `wrangler.jsonc`: Cloudflare deployment config copied into generated output.

## Content-Facing Components

Agent Sitegen discovers component metadata from `@sitegen` comments inside `src/components/*.astro`.

Example:

```astro
---
/**
 * @sitegen
 * type: feature-block
 * description: Two-column feature section with text, media, diagram, and optional CTA.
 * group: Content Sections
 */
const { heading, body } = Astro.props;
---
```

Use these comments for component-level metadata:

- `type`: section type used in `content/site.yaml`
- `description`: what the section is for
- `group`: how the component should be grouped in generated prompts
- `props`: optional prop hints for LLMs and future validation

Use `sitegen.template.json` for template-level metadata:

- template name and description
- generated file paths
- component groups
- private or excluded helper components
- explicit component overrides

## Adding A New Section Type

When a page needs a new reusable section:

1. Add a new Astro component in `src/components/`.
2. Give it a stable PascalCase filename, such as `LogoCloud.astro`.
3. Add an `@sitegen` comment with a kebab-case `type`, such as `logo-cloud`.
4. Implement the component using `Astro.props`.
5. Add the new section to `content/site.yaml`.
6. Run `npm run validate`.
7. Run `npm run generate`.
8. Run `npm run --prefix ./site build`.

Keep the component reusable. Do not hard-code one page's copy into the component.

## Private Helper Components

Some components are implementation helpers, not content section types. Keep those listed in `privateComponents` in `sitegen.template.json` so LLMs do not choose them for `site.yaml`.

Examples:

- `button`
- `cta-button`
- `navbar`
- `footer`
- `section-shell`
- `metric-display`
- `feature-card`

If a helper becomes a real content section, remove it from `privateComponents` and add an `@sitegen` comment.

## Deployment Notes

This template is intended to generate a Cloudflare-compatible static site.

Keep deployment files in the template so every generated site receives them:

- `wrangler.jsonc`
- `package.json`
- `astro.config.mjs`
- `public/.assetsignore`

If Cloudflare reports a missing file, fix it in this template and regenerate the site.

For Cloudflare Workers Git integration, deploy this as a static assets Worker. The build creates `dist/`, then Wrangler deploys it using `wrangler.jsonc`.

If the Cloudflare project root is the generated `site/` folder:

```txt
Build command: npm run build
Deploy command: npx wrangler deploy
```

If the Cloudflare project root is this repository root:

```txt
Build command: npm run build
Deploy command: cd site && npx wrangler deploy
```

`wrangler.jsonc` must point Workers at the Astro static output:

```json
{
  "assets": {
    "directory": "./dist"
  }
}
```

The manual deploy script also uses Cloudflare Workers:

```bash
npm run cf:deploy
```

## Diagram Exports

Diagram components can expose `data-export-id` attributes. The root script `scripts/export-diagrams.mjs` captures those elements from the built site and writes PNG files to `scripts/exports/`.

Those exports are not required for the site build unless a page explicitly references them.
