Understanding Stigmergy Coordination

Nature's most efficient coordination mechanism, applied to software development.

What is Stigmergy?

Stigmergy is indirect coordination through environmental modification. In nature, ants leave pheromone trails that other ants follow, enabling complex coordination without direct communication or central control.

Codebolt applies this principle to software development. Agents leave digital "pheromones" on tasks that signal state, intent, and needs. Other agents perceive these signals and act accordingly, creating self-organizing intelligence.

Example: Overnight Development

5:30 PM - You Set Priorities

You deposit "importance" pheromones on urgent tasks before leaving.

6:00 PM - Swarm Activates

Agents sense high-importance signals and naturally gravitate to urgent tasks.

8:00 PM - Agent Discovers Complexity

Agent finds task too large, deposits "request_split" pheromone.

8:15 PM - Split Proposals

Multiple agents propose splits. Deliberation selects best approach.

10:00 PM - Parallel Execution

New tasks get "job_available" pheromones. Agents distribute to sub-tasks.

2:00 AM - Blocking Issue

Agent deposits "task_not_ready" pheromone when blocked.

3:00 AM - Blocker Resolved

Another agent resolves dependency, changes to "job_available."

9:00 AM - You Review

Wake up to completed features, tested code, and pull requests ready for merge.

Common Development Workflows

🔄 Single Agent Task Execution

Use Case: Quick coding assistance for specific task

Open Chat Panel → Agent ready
Describe task → Agent analyzes codebase
Wait for execution → Agent codes, tests, commits
Review in Code Panel → Changes highlighted
Accept or request edits → Agent iterates if needed
Merge changes → Task complete

⚡ Parallel Feature Development

Use Case: Develop multiple features simultaneously

Define tasks in Jobs Panel
Deploy swarm with 5+ agents
Agents work on Feature A, B, C in parallel
Each agent codes + tests independently
Review in ReviewMergeRequestPanel
Merge approved changes

🌙 Overnight Development

Use Case: Agents work while team sleeps

Evening: Define jobs, set priorities, configure swarm
Overnight: 10-50 agents work on features, bugs, tests, docs
Automatic: PRs created via Review Merge Request Panel
Morning: Review PRs, check tests, merge approved

🗳️ Agent Deliberation

Use Case: Agents make consensus decisions

Agent creates deliberation: "Best approach for refactor?"
Other agents submit proposals
Agents vote on responses
Winner determined by vote count or reputation-weighting
Winning approach implemented

🔍 Test-Driven Development

Use Case: Agent writes tests first, then implementation

User specifies feature requirements
Agent writes comprehensive tests
Tests committed (failing)
Another agent implements to pass tests
Cycle continues until all tests pass
Review and merge
